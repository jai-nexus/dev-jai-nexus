# Challenge: Bounded Dispatch and Coverage Integration Spec v0

**Motion:** motion-0091
**Challenger role:** challenger
**Date:** 2026-03-30

## Review

Both artifacts are structurally sound and consistent with WS-A through
WS-D. Two objections raised.

---

### C-1: coverage-declaration.schema.yaml duplicates fields already in
agency.yaml — is it necessary?

**Concern:** `config/agency.yaml` (Wave 0 bootstrap artifact) already
contains NH IDs, roles, and scope information. The coverage declaration
appears to restate much of that data. Why add another artifact?

**Resolution:** The coverage declaration serves a different purpose than
agency.yaml. `agency.yaml` is a runtime configuration file for the council
runtime — it is optimized for the agent runtime reader, not for an external
integrator. The coverage declaration is a planner-owned summary optimized
for dispatch routing and wave tracking. Key differences: (1) it adds
`wave_state` which agency.yaml has no concept of; (2) it adds provenance
annotations; (3) it aggregates governance surface paths from multiple
bootstrap artifacts into one document; (4) it is authored and owned by the
operator at Wave 0 close, not generated by the bootstrap tool. The
duplication is intentional aggregation, not redundancy. Accepted.

---

### C-2: The planner/implementation boundary diagram implies the planning
system "freezes" fields — but who enforces this freeze?

**Concern:** The spec states that planner-owned fields "must not be mutated
by runtime code without a governed motion" but provides no enforcement
mechanism. This is aspirational, not contractual.

**Resolution:** Enforcement is governance-procedural, not technical. The
project constitution (Wave 0 artifact) documents the governed change process.
A runtime system that silently mutates a planner-owned field like
`governance_resident_repo` would be violating repo governance, not just a
spec guideline. The freeze rule is correctly classified as a governance
constraint. Specifying a technical enforcement mechanism is out of scope for
a planning spec. Accepted.

---

## Verdict

No blocking objections. Both resolved. Motion-0091 spec artifacts are valid.
The motion-0084 planning program is complete through WS-E.
