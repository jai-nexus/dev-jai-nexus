## What WS-C derives vs what WS-A already captures

- `topology.shape` (monorepo | polyrepo)
- `topology.repos[]` (name, domain_scope, tier, governance_resident)

WS-C derives and adds:

| Field | Rule |
|---|---|

| `cross_repo_links[]` | Inter-repo dependency edges (polyrepo only; optional) |

## Governance-resident repo: operational definition

In the topology planner, "governance-resident repo" means the repo that:

1. Hosts `.nexus/` (motions, context, planning directories)
2. Hosts `config/agency.yaml`
3. Is the scope target for all 5 governance agents
4. Is the scope target for OPERATOR and LIBRARIAN (by WS-B default)
5. Is the repo where Wave 0 bootstrap artifacts are emitted

For monorepo: always the single repo (implicit).
For polyrepo: exactly one repo from `topology.repos` where `governance_resident=true`.

The governance-resident repo is NOT required to be the repo where most
execution work happens — it is the repo where governance state lives.

## Repo profiles

Per-repo profiles enrich the intake `topology.repos[]` with planning-time
data that WS-D needs to emit correct agency.yaml scope entries.

**Key profile fields:**

- `agent_scope_paths[]` — path patterns this repo contributes to agent scope
  (e.g., `paths:portal/src/**`, `paths:.nexus/**`). Grounded in agency.yaml
  baseline pattern from dev-jai-nexus.
- `roles_scoped_here[]` — execution roles that include this repo in `scope_repos`.
  Derived from WS-B `governance_resident_only` flags.
- `governance_resident` — echoed from intake for downstream consumers.

**Agency.yaml path pattern convention** (from dev-jai-nexus baseline):

| Repo area | Pattern example |
|---|---|
| App source | `paths:portal/src/**` |
| Scripts | `paths:portal/scripts/**` |
| Docs | `paths:portal/docs/**` |
| Config | `paths:config/**` |
| Governance | `paths:.nexus/**` |
| Deny | `deny:portal/.env*`, `deny:**/*.env*`, `deny:**/secrets/**` |

## Wave model

Five bootstrap waves grounded in the dev-jai-nexus completed baseline.

### Proof-based advancement

Wave N+1 requires at least one completed proof from Wave N. "At least one"
means a single instance suffices — exhaustive coverage is not required for
wave advancement. This matches the dev-jai-nexus baseline: the canonical
reference path (packet 880 / motion-0070) proved the full loop end-to-end
with a single instance before closure.

### Wave definitions

**Wave 0 — Governance Substrate**

Responsibility: Establish the minimum governance infrastructure.

Must emit (auto-generated by WS-D from intake + demand output):
- `config/agency.yaml` — from demand matrix
- `.nexus/context/project-constitution.yaml` — from intake
- `.nexus/motions/motion-0001/motion.yaml` — stub only

Must be manually authored:
- `.nexus/motions/motion-0001/proposal.md` — inaugural governance decision

Copied from dev-jai-nexus baseline (no modification):
- `.nexus/context/slot-policy.yaml`
- `.nexus/context/scoring-rubric.yaml`
- `.nexus/context/motion-packet.schema.json`

Proof gate (minimum):
- `config/agency.yaml` committed and parseable
- Council agent entry present in agency.yaml
- Inaugural motion package committed (motion.yaml + proposal.md minimum)

**Wave 1 — Planning Activation**

Responsibility: Prove governed planning works.

Activities:
- First architect agent runs against an open motion
- First work packet issued and motion-tagged
- Council runs and produces a decision

Proof gate (minimum):
- At least one motion reaches `status: RATIFIED` in `decision.yaml`
- At least one work packet exists with a `motion:<motionId>` tag

**Wave 2 — Execution Activation**

Responsibility: Prove the full execution chain works.

Activities:
- Builder + verifier claim and complete execution work
- Queue bridges operational
- Full ARCHITECT → BUILDER → VERIFIER chain completed in a single packet

Proof gate (minimum):
- At least one packet has ARCHITECT evidence (debug.plan or equivalent)
- AND builder evidence (debug.patch, PR, or patch artifact)
- AND verifier evidence (debug.verify or verification artifact)
- All three in the same motion-linked packet

**Wave 3 — Operator Surface**

Responsibility: Prove operator-surface closure works.

Activities:
- Governing motion section visible on operator work detail
- Operator takes a decision action (APPROVE / REQUEST_CHANGES / REQUEUE)
- `execution.receipt.json` written to `.nexus/motions/<motionId>/`

Proof gate (minimum):
- At least one `execution.receipt.json` exists in a motion directory
- `status: COMPLETED` (or CHANGES_REQUESTED / REQUEUED) in the receipt

**Wave 4 — Coherence and Coverage**

Responsibility: Prove the full governed loop is coherent.

Activities:
- Loop coherence gate evaluated for a completed packet
- Coverage declarations made

Proof gate (minimum):
- At least one packet returns `verdict: COHERENT` from the loop coherence gate
- All five conditions met: RATIFIED → ISSUED → executed → approved → COMPLETED

### Wave advancement: monorepo vs polyrepo differences

**Monorepo:**
- Waves 0–4 execute entirely within one repo
- All proof gates are repo-local
- No cross-repo coordination required at any wave

**Polyrepo:**
- Wave 0: governance substrate in governance-resident repo only; secondary
  repos do not require any Wave 0 artifacts
- Wave 1: planning activation in governance-resident repo; architect agent
  scope spans all repos but first proof may be governance-resident only
- Wave 2: execution chain should include at least one packet where builder
  or verifier touches a secondary repo path (to prove cross-repo scope works)
- Wave 3: receipt closure in governance-resident repo (OPERATOR is scoped
  there by WS-B default)
- Wave 4: coherence evaluation is governance-resident-repo-anchored

## OffBook.ai pressure-test

**Topology plan output for OffBook.ai:**

```
project_id: offbook-ai
execution_scope: cross_repo          # derived from polyrepo shape
governance_resident_repo: offbook-core

repo_profiles:
  - name: offbook-core
    tier: primary
    governance_resident: true
    roles_scoped_here: [council, proposer, executor, challenger, arbiter,
                        ARCHITECT, BUILDER, VERIFIER, OPERATOR]
    agent_scope_paths:
      - paths:src/**         # API source (domain_scope: api)
      - paths:.nexus/**
      - paths:config/**
      - deny:**/.env*
      - deny:**/secrets/**

  - name: offbook-web
    tier: secondary
    governance_resident: false
    roles_scoped_here: [ARCHITECT, BUILDER, VERIFIER]
    agent_scope_paths:
      - paths:src/**         # frontend source (domain_scope: frontend)
      - deny:**/.env*
      - deny:**/secrets/**

cross_repo_links:
  - from_repo: offbook-web
    to_repo: offbook-core
    relationship: api_consumer
    notes: "Web client consumes core booking/availability API."
```

**Wave sequence for OffBook.ai:**

Wave 0 proof: agency.yaml committed to offbook-core, inaugural motion committed.
Wave 1 proof: first motion ratified in offbook-core; first work packet issued.
Wave 2 proof: one packet with ARCHITECT/BUILDER/VERIFIER chain; builder or
verifier touches offbook-web path to prove cross-repo scope.
Wave 3 proof: one execution.receipt.json in offbook-core/.nexus/motions/.
Wave 4 proof: one packet returns COHERENT from coherence gate.

Fits cleanly. No ambiguities.
