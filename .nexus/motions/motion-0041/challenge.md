# Challenge - motion-0041

## Main concern
A code-heavy live usage proof can fail in two different ways: either the task is too trivial to stress the setup, or it is too broad and the session result becomes noisy and hard to judge. If the task choice is poor, the motion will not produce useful evidence.

## Risks
1. **Too-small-task risk**
   - The task may be so small that it does not meaningfully test code-context setup.

2. **Too-broad-task risk**
   - The task may require too much repo context, making the result difficult to interpret.

3. **Attribution risk**
   - Success may be credited to the setup even if most useful context came from manual re-briefing.

4. **Noise risk**
   - The report may become a loose narrative instead of recording the actual inputs and missing pieces.

5. **Artifact confusion**
   - Session notes may blur the distinction between canonical code/motion artifacts and generated operational artifacts.

## Required safeguards
- choose one real bounded code task,
- record the exact setup inputs used,
- record the exact additional code surfaces needed,
- record what still required manual explanation,
- keep the output focused on operational learning.

## Challenger standard for approval
I approve this motion only if:
- one real code-heavy task is actually attempted,
- the setup inputs are recorded concretely,
- the motion package remains central,
- the resulting lessons are specific enough to improve future Claude code sessions,
- the motion remains bounded and evidence-oriented.
