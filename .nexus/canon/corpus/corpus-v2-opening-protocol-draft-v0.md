# Corpus V2 Opening Protocol Draft v0

## Purpose

Draft the future protocol for opening Corpus V2 without actually opening it.

This artifact is protocol-only and does not open Corpus V2 here.

## Opening preconditions

A future Corpus V2 opening motion must prove:

- readiness checklist is satisfied or explicitly waived with visible rationale
- gate evidence is current
- `corpus_id` is assigned
- numbering reset path is defined
- Corpus V1 immutability is preserved
- agent governance path is operational
- authority boundaries are explicit

## Required proof and evidence

The future opening motion should provide:

- current readiness checklist evidence
- current gate-model evidence where applicable
- current schema-valid motion package evidence
- current snapshot evidence
- explicit human approval boundary
- explicit statement of any waived or deferred gate

## Required output artifacts

A future opening motion should produce, at minimum:

- Corpus V2 model artifact
- initial `motion-0001` package
- snapshot namespace or corpus-aware snapshot treatment
- operator visibility update

These outputs belong to the future opening motion, not to this draft.

## Control-thread decision requirements

The future `CONTROL_THREAD` opening decision should:

- confirm readiness evidence is current
- confirm any waived gate is explicitly accepted
- confirm numbering reset treatment is safe
- confirm Corpus V1 remains preserved and searchable
- confirm authority boundaries remain explicit

## Not opened here

This draft does not:

- open Corpus V2
- reset numbering
- create `motion-0001`
- mutate Corpus V1 records

## Non-goals

This protocol draft does not:

- implement live drafting
- implement live voting
- implement autonomous ratification
- add provider or model calls
- add runtime execution
- add branch-write, PR, merge, scheduler, or automation authority

## Authority boundary

This opening protocol draft preserves the settled authority boundary:

- no provider or model calls
- no Agent runtime execution
- no branch-write authority
- no PR-creation authority
- no merge authority
- no scheduler
- no automation
- no API or DB mutation
- no hidden persistence
